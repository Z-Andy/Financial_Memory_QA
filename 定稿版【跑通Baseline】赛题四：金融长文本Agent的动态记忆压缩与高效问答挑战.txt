1. 赛题速览
1.1 任务定义
赛道四的核⼼任务是⾦融⻓⽂本问答。出题⽅提供五类⾦融⽂档和配套的选择题，要
求参赛者构建 Agent 系统，在控制 Token 消耗的前提下准确作答。
赛事官⽅地址：https://tianchi.aliyun.com/competition/entrance/532486
⼤部分题⽬涉及跨⽂档对⽐、公式计算、条款推理和逻辑判断，需要系统具备信息定
位、知识理解和推理计算三层能⼒。
1.2 关键约束
• 模型限制：正式提交只准调⽤ Qwen 系列模型 API
• 题⽬范围：A 榜题⽬附带 doc_ids，系统只需在指定⽂档中定位答案，不需要全
⽹搜索
• 答案格式：单选（mcq）、多选（multi）、判断（tf）三种
• 答案标准化规则：单选题和判断题以⾸个有效答案字⺟为准；多选题将答案字⺟
去重并排序后做完全匹配，不设置部分分，漏选、错选、多选均计为错误
• Token 预算：500 万 Token 总量限制，评分兼顾准确率与 Token 效率
1.3 评分机制：准确率优先
最终得分公式⼤致为 准确率 × (0.7 + 0.3 × Token 效率分)。准确率权重占 70%，
Token 效率占 30%。这意味着：即使你把 Token 压得极低，准确率上不去，总分也⾼
不了；反之，准确率够⾼，Token 稍微超⼀点，对总分的影响有限。因此优化的第⼀
优先级是准确率，第⼆优先级才是 Token 效率。

1.5 赛题特点解析
做这道题时，我们发现⼏个容易被低估的难点：
第⼀，doc_ids 只是范围提示，不是答案指针。 题⽬告诉你看哪⼏份⽂档，但答案不
会直接写在⽂档第⼀段。很多时候需要把 4 份条款的公式串起来计算，或者对⽐两份
年报的同⼀指标才能判断对错。
第⼆，⾦融知识⻔槛会过滤掉模型的常识推理。 保险条款中的"等待期""免赔额""责任
免除"，监管法规中的"受益所有⼈识别""客户尽职调查"，这些概念有严格的法律定义
和适⽤条件。模型如果没有精准定位到对应条款，凭"常识"推断很容易选错。
第三，题型之间差异⼤，没有统⼀解法。 计算题需要提取公式后代⼊数值；跨⽂档对
⽐题需要同时打开多份⽂档提取同⼀类数据；条款定位题需要精准找到"责任免除""等
待期"等特定章节；事实查找题需要在⼏万字的研报中定位⼀个带年份限定的数字。⽤
同⼀套策略处理所有题型，效果必然打折。
1.6 思维链（CoT）的实验发现
我们在实验中也测试过让模型"先推理再回答"的 CoT 策略：在 prompt 中要求模型对
每个选项引⽤⽂档条款、说明判断依据，最后再给出答案。

效果是两极分化：
• 对复杂题有提升。⽐如跨⽂档对⽐题和多选题，模型被迫逐条验证，漏选和误选
明显减少。部分计算题在 CoT 引导下，提取公式的准确率也有改善。
• Token 消耗暴涨。baseline 的平均 completion token 只有 2-5 个（模型直接输出
字⺟），开启 CoT 后飙升到⼏百个。100 题跑下来，completion token 可能从 15 万涨
到 100 万以上，对 500 万总预算的消耗⾮常可观。
结论：CoT 可以按需使⽤，不建议全量开启。在预算有限的前提下，更合理的策略是"
按需开启"，只在计算题、多选题、跨⽂档对⽐题等复杂题型上启⽤ CoT，简单的事实
查找题直接输出答案。这也是后续优化的⼀个⽅向。但是我建议都不要开，⾮必要的
话。
1.7 关于框架选择
⼀个常⻅的参赛误区是：上来就⽤ LangChain、LlamaIndex 等现成框架搭建 Agent。
这些框架为了通⽤性，在 prompt 模板、⼯具调⽤协议、中间状态记录上做了⼤量封装，
带来的直接后果是 Token 消耗会稍⾼于⾃定义设计的流程，如果是⼤家都能获得答题
准确率满分的背景下，后⾯⼤家应该是需要组⾃⼰设计如何组装prompt会更节省
token的。
从评分公式看，准确率占 70%、Token 效率占 30%，两者都不是可以忽略的⼩头：
• 如果参赛者的准确率都能做到不错（⽐如 60% 以上），这时排名差距主要来⾃
Token 效率。谁能在同等准确率下消耗更少的 Token，谁就能拿到更⾼的效率分。
• 如果整体准确率都不⾼（⽐如普遍低于 40%），这 时 排 名差距主 要 来⾃ 准确率。
Token 压得再低，准确率上不去也没有意义。
真正榜单前列的选⼿，⼀定在准确率和 Token 消耗上做到了平衡。 框架可以帮你快速
搭建原型，但在正式提交前，建议把核⼼流程拆开重写，去掉不必要的 prompt 冗余和
中间层封装，把 Token 预算⽤在最关键的地⽅。
赛题详细说明⻅ 赛题与数据.md。

为什么只⽤简单截断，不做检索
很多参赛者第⼀反应是"⽂档太⻓，应该先搜再读"。这个思路本⾝没错，但过早引⼊检
索会带来⼀系列连锁问题。核⼼⽭盾在于：检索必然引⼊切⽚，⽽切⽚会破坏原⽂的
连贯性。
⼀段保险条款可能横跨三个⾃然段：前⼀段定义概念、中间⼀段给出公式、后⼀段附
加例外条件。如果把这三段切成独⽴的⽚段分别检索，模型看到的可能只是"现⾦价值
= 累计保费 × 75%"，却看不到后⾯紧跟着的"因意外伤害导致的退保除外"。这种连贯
性断裂是检索⽅案的根本⻛险。
从信息连贯性的⻆度看，全⽂ > 摘要 > 检索⽚段。全⽂让模型⾃⼰决定哪⾥重要，上
下⽂天然完整；摘要虽然损失了细节，但⾄少保持了逻辑链条；检索⽚段则是最脆弱
的，召回结果的质量直接决定了最终答案的质量，⽽单⼀检索策略（纯 BM25 或纯向
量搜索）的召回质量往往不稳定。
RAG 系统⾥之所以要设计"粗排 + 细排"的两阶段架构，正是因为粗排（BM25 或向量
相似度）召回的结果中，经常夹杂着不相关或弱相关的内容，甚⾄因为词频巧合、
embedding 漂移等各种原因把⽆关⽚段排得很靠前。细排（rerank）的作⽤就是做⼆
次筛选，把真正相关的⽚段提上来。但在baseline阶段，我们没有⾜够的调优经验来
确定阈值、去重策略和融合权重，贸然引⼊检索反⽽可能让模型⾯对⼀堆零散、重复、
缺上下⽂的⽚段，理解成本⽐直接读截断全⽂还⾼。
另外，检索不是加了就有效果的模块。它需要配合滑动窗⼝做⽚段边界保护、需要多
路召回融合来互补 BM25 和向量搜索的盲区、需要动态阈值来过滤低质量⽚段。这些
配套机制没有到位时，检索的提分潜⼒⽆法释放。
当然，直接截断也有⾃⼰的问题（前半截可能刚好没有答案），但这属于"信息够不够"
的问题，⽐"信息对不对、连不连贯"更容易排查和迭代。baseline 阶段先固定⼀个稳定
的上限，跑通端到端流程，后续再逐步引⼊检索并验证真实收益，是更务实的策略。

⼀个参考基准：赛题裸跑是什么⽔平
作为参照，赛题基于 Qwen-plus 做了⼀个最简单的基线：把⻓⽂档直接输⼊模型、不
做检索也不做优化，A 组 100 题答对 17 道，准确率约 17%，Token 消耗约 362 万。
这说明：
• 本赛题确实不容易，裸跑模型只有约 17% 的准确率
• 本 baseline 的 16.62 分和这个裸跑基准处于同⼀⽔平，没有拖后腿
• 从零到 17 分不需要什么技巧，但从 17 分往上提分，每⼀步都需要精⼼设计
因此本 baseline 的策略是：先把端到端流程跑通，⽤⼀个稳定的基准来衡量后续改动
的真实收益。如果你改了检索策略但分数没涨，⾄少知道不是基础流程的问题。
它不做的事（留给后续版本）：
• 没有检索模块，直接截断⽂档前半部分
• 没有多轮推理，单轮提问、单轮回答
• 没有证据链追溯，⽆法解释为什么选某个答案
• 不做题⽬分类，所有题型⽤同⼀套 prompt

为什么选择 no-tool ⽽⾮ Function Calling
我们也测试过带 Function Calling（FC）的版本。FC 需要在每次请求时把⼯具声明
（schema）放⼊ system prompt，让模型决定什么时候调⽤⼯具、传什么参数。实测
A 组 100 题的 Token 消耗对⽐如下：
指标 no-tool tool（FC） 差异
prompt tokens 198,738 273,226 +37.5%
completion tokens 154,294 142,726 -7.5%
total tokens 353,032 415,952 +17.8%
FC 的 completion tokens 确实略少（因为输出被结构化成了 JSON）， 但 prompt
_tokens 多了近 40%。原因是每道题的 prompt 都要携带⼯具声明，⽽且模型需要先⽣
成 tool call 的 JSON，再等待外部函数执⾏结果，再发起第⼆轮请求。整个流程的
Token 开销和交互复杂度都更⾼。
本 baseline 采⽤ no-tool ⽅案，⽤更少的 Token 完成同⼀批题⽬，流程也更简单：读
⽂档、拼 prompt、直接出答案，没有中间状态。

ContextManager：简单截断

class ContextManager:
def __init__(self, max_chars: int = 320000, max_doc_chars: int
= 4000):
self.max_chars = max_chars # 总 prompt 上限
self.max_doc_chars = max_doc_chars # 单篇文档上限
def truncate_doc(self, text: str) -> str:
if len(text) <= self.max_doc_chars:
return text
return text[:self.max_doc_chars] + "\n\n[文档后续内容已省
略]"
max_doc_chars=4000 是⼀个经验值。保险/监管⽂档通常 5000-8000 字符，截断到
4000 基本能覆盖核⼼条款；财报/研报动辄 10 万字符以上，4000 字符只能覆盖⽬录
和前⾔，这是本 baseline 最⼤的短板。

class PromptBuilder:
def build_prompt(self, question, evidence, context_manager):
# 每篇文档单独截断后拼接
context_parts = []
for ev in evidence:
content = context_manager.truncate_doc(ev.content)
context_parts.append(f"【文档 {ev.doc_id}】
\n{content}")
context = "\n\n".join(context_parts)
options_text = "\n".join(
[f"{k}. {v}" for k, v in question.get("options",
{}).items()]
)
prompt = (
"你是一位金融文档分析专家。请根据以下提供的文档内容，回答问
题。\n"
"要求：\n"
"1. 仔细阅读文档中的相关条款、数据和事实\n"
"2. 对每个选项进行独立判断\n"
"3. 只输出最终答案字母，不要解释过程\n"
"4. 多选题答案按字母顺序排列，不加分隔符\n\n"
f"{context}\n\n"
f"问题：{question['question']}\n\n"
f"选项：\n{options_text}\n\n"
f"答案："
)
return prompt
这个 prompt 的设计思路：
• 明确⻆⾊（⾦融⽂档分析专家），让模型进⼊"严肃阅读"模式
• 要求"只输出字⺟"，减少后处理难度
• 多选题要求"按字⺟顺序排列"，避免 "AC" 和 "CA" 被判定为不同答案

7. 局限与优化路线图
本 baseline 的每⼀个局限都对应⼀条清晰的优化路径。但在动⼿优化前，需要建⽴⼀
个正确预期：本次赛题的准确率并不容易提升。
原因有三：
1. 题型对模型综合能⼒要求⾼。⼀道题可能同时涉及跨⽂档阅读（4 份条款）、公式
计算（提取参数后代⼊）、逻辑推理（判断例外情形是否适⽤）和精细⽐较（排序或判
断正误）。这意味着模型不仅要"读到"信息，还要"算对""想通""⽐准"。
2. ⾦融知识本⾝⻔槛⾼。保险条款中的"等待期""免赔额""责任免除"，监管法规中的"
受益所有⼈识别""客户尽职调查"，这些概念对⾮专业⼈⼠来说理解成本很⾼。模型如
果没有⾜够的⾦融领域预训练知识，光凭 prompt 很难做到精准理解。
3. 优化⽅案往往有副作⽤。⽐如引⼊检索可以提升⻓⽂档覆盖率，但可能漏掉关键
否定词；让模型做详细推理可以提升判断准确性，但会⼤幅增加 Token 消耗。每改⼀
个地⽅，可能解决⼀个问题，⼜引⼊另⼀个问题。
因此，不要期望某个单⼀技巧就能让准确率⼤幅提升。真实的提分路径是多模块协同
优化：⽂档解析质量 + 检索质量 + 上下⽂管理（动态记忆压缩）+ prompt 设计 + 答案
检查，层层叠加才能看到效果。
与官⽅优化⽅向的对照
赛题官⽅在 baseline 说明中明确列出了五个重点优化⽅向，和我们的思路基本⼀致：
官⽅⽅向 对应本教程的模块
⽂档解析质量：提升 PDF ⽂本抽取、表 8.1 PDF 解析质量

7.1 PDF 解析质量
本 baseline 使⽤ pymupdf4llm 将 PDF 转为 Markdown，解析质量直接决定了后续所
有环节的上限。如果解析阶段出现以下问题，准确率再⾼也救不回来：
• 表格丢失或错位：财报中的三⼤报表如果⾏列对应关系被破坏，模型读到的"营业
收⼊"和"净利润"可能是同⼀列的数据，计算题直接全军覆没
• 数字截断或变形：条款中的⾦额、⽐例、年限等关键数字如果解析出错（⽐如
"100,000"变成"100"），公式计算的 结果必然错误
• 段落顺序错乱：保险条款中"等待期"的定义和"例外情形"如果不在同⼀段落附近，
模型可能只看到前半句"等待期内不赔"，漏掉后半句"因意外伤害除外"
• 标题层级丢失：监管法规的"第⼀章 总则""第⼆章 适⽤范围"等层级结构如果被打
平成纯⽂本，模型难以判断条款的适⽤优先级
优化⽅向：
• 多解析器对⽐：同⼀篇 PDF ⽤ pymupdf4llm、Marker、MinerU 等多种⼯具解析，
取质量最好或互补性最强的⼀份
• 表格专项处理：对财报/合同中的表格使⽤专⻔的表格解析⼯具（如 camelot、
tabula），保 留 ⾏ 列 结 构， M转arkd成own 表格或 CSV
• 版⾯结构还原：在 Markdown 中保留⻚码、章节标题、条款编号等结构信息（如
# 第一章 保险责任、**第 3.2 条 等待期**），帮助模型快速定位
• 解析后校验：⽤规则或 LLM 检查解析结果是否存在明显异常（如连续 20 ⾏都是
乱码、表格列数不⼀致、⾦额数字明显缺失），异常时切换备⽤解析器
需要特别注意的是：PDF 解析是⼀次性投⼊。A 榜 86 份 PDF 解析⼀次后可以复⽤，
不受 500 万 Token 总预算限制。在解析阶段投⼊精⼒换取更⾼质量的输⼊⽂本，性价
⽐远⾼于在答题阶段⽤更多 Token 去弥补解析错误。

7.2 上下⽂管理粗放（⽂档截断粗暴）
本 baseline 保留⽂档前 4000 字符，对于财报/研报/合同来说命中关键信息的概率很低。
这是上下⽂管理问题的⼀个具体表现：系统对"保留什么、丢弃什么、怎么拼接"没有任
何策略，只是机械地截断。优化⽅向分两层：
（1）⽂本切⽚ + BM25
先把⽂档切成固定⻓度或按段落切分，再⽤ BM25 算法计算每个⽚段与问题的相关性
得分，取 top K 送给 LLM。优点是轻量、不依赖外部模型，jieba 分词 + rank-bm25 库
就能跑通；缺点是只考虑词频匹配，⽆法理解语义（⽐如"退保费⽤"和"退保⼿续费"会
被当成两个词）。
（2）⽆索引关键词匹配 + 滚动窗⼝
不建索引，直接在⽂档中搜索问题关键词，提取关键词所在位置的前后若⼲字符作为
上下⽂。复现⻔槛最低，但召回质量波动⼤。
对应实现⽅式：
• BM25 单次检索 top 5：⽤ jieba 分词、按领域差异化 chunk_size，轻量且可控
• ⽆索引关键词匹配 + 滚动窗⼝：复现⻔槛更低，不依赖预建索引
上下⽂管理的更⾼层次：⽚段组装与动态压缩
即使有了检索，如何把搜到的⽚段拼成⼀份⾼质量的 prompt，仍然是上下⽂管理的核
⼼问题：
• ⽚段去重与边界修复：检索返回的 top K ⽚段可能存在重叠（⽐如 chunk1 结尾和
chunk2 开头重复了 200 字），直接拼接会造成内容冗余，浪费 Token。需要去重并保
证段落边界完整。
• 信息密度排序：检索结果不是全部值得放⼊ prompt。按相关性得分排序后，设定
⼀个阈值，只保留得分明显⾼于阈值的⽚段，把有限的上下⽂窗⼝留给真正相关的信
息。
• 多轮上下⽂传递：迭代检索时，第⼀轮搜到的关键信息要在第⼆轮 prompt 中保留，
避免"搜到了⼜忘了"。但同时要控制历史信息的累积⻓度，防⽌ prompt ⽆限膨胀。
• 按题型动态调整上下⽂⻓度：简单的事实查找题可能只需要 2000 字的上下⽂；跨
⽂档对⽐题可能需要 4 篇⽂档各取 top 3 ⽚段，总共上万字；计算题则需要确保公式
所在的段落不被截断。上下⽂⻓度不应是固定值，⽽应根据题⽬复杂度⾃适应调整。

7.3 单轮推理⽆⾃检、答案检查薄弱
本 baseline 只做单轮调⽤，模型给出答案后不做⼆次核验，对错全凭⼀次输出。⽽且
答案检查仅限于格式校验（是不是合法字⺟），不做内容层⾯的合理性判断。优化⽅向
分两层：
（1）推理层：从单轮到多轮
• 迭代检索：LLM 判断信息是否充分，不够则⽣成新查询继续搜
• ⾃我纠错：让模型回顾⾃⼰的答案，检查是否与⽂档证据⽭盾。⽐如模型选了 A，
但⽂档明确说"以下情形不属于保险责任"，⾃我纠错环节可以发现这个冲突
• 多选题逐项验证：多选题让每个选项单独过⼀遍"⽂档是否⽀持"，⽐⼀次性让模型
选所有答案更不容易漏选
（2）答案检查层：从格式校验到内容校验
当前 baseline 的 AnswerValidator 只检查答案格式（mcq 是不是单个字⺟、multi 字
⺟是否排序），这是最底层的检查。更⾼层的检查包括：
• 证据⼀致性校验：模型说答案是 A，但检索到的证据⽚段明确⽀持 B，系统应该能
识别这个⽭盾。实现⽅式可以是让模型在输出答案的同时给出判断依据，然后⽤另⼀
轮调⽤验证"依据是否⽀持答案"
• 题型规则校验：判断题只有 A/B 两个选项，如果模型输出 C，直接判定为异常并
fallback；多选题⾄少要有两个正确选项（按出题习惯），如果模型只选了⼀个，可以
触发⼆次确认
• 多轮⼀致性校验：同⼀道题⽤相同 prompt 调⽤两次，如果两次答案不同，说明模
型对该题不确定，可以引⼊第三次调⽤取多数，或改为保守策略（如多选题在有争议
时少选）
• 位置偏差修正：如果发现某次提交的答案 A 明显偏多（如 40% 以上），可以对 A
选项的题增加⼆次确认，或在 prompt 中加⼊"不要偏向第⼀个选项"的去偏指令
• 默认值策略：模型输出为空或⽆法解析时，按题型规则 fallback。单选默认 A，判
断默认 A，多选默认 AB（保守策略是选最少的可能组合）
7.4 Prompt 简陋
本 baseline ⽤同⼀套 prompt 处理所有题型和领域。优化⽅向：
• 混合策略：短⽂档全⽂读取、⻓⽂档检索，按⽂档⻓度动态选择策略
• 领域特化：针对不同领域和题型设计专⽤提示词，加⼊去偏指令（"不要偏向第⼀
个选项"）

7.5 ⽆结构化知识提取
本 baseline 把⽂档当纯⽂本处理，模型每次回答都要重新阅读原始段落。如果题⽬要
求⽐较 4 份保险条款的等待期、免赔额、退保费⽤，模型需要在 4 份⽂档中反复搜索
同类信息，效率低且容易遗漏。
优化⽅向：构建领域知识图谱
知识图谱的核⼼价值在于把分散在各⽂档中的结构化信息抽取出来，统⼀存储为"实体-
关系-属性"三元组。例如从保险条款中提取：
Plaintext
(国寿增益宝, 等待期, 30 天)
(国寿增益宝, 免赔额, 10000 元)
(国寿增益宝, 第7年退保费用, 0%)
(平安智盈金生, 等待期, 90 天)
...
当题⽬问"以下哪款产品的等待期最短"时，系统不需要再去读原始⽂档，直接从图谱中
按"等待期"属性排序即可。这种⽅式在跨⽂档聚合类题⽬上优势明显：信息已经结构化，
⽐较和推理的准确率远⾼于让模型在原始⽂本中⾃⾏定位。
具体实现可以分两步：
1. 离线抽取：预处理阶段⽤ LLM 遍历所有⽂档，按领域定义的信息模板（保险条款
提取"等待期/免赔额/退保费⽤/⾝故保险⾦"，财报提取"营收/净利润/研发投⼊/现⾦流"）
抽取结构化字段，存⼊ JSON 或图数据库
2. 在线查询：答题时先判断题⽬是否涉及结构化⽐较，如果是则优先查知识图谱；
如果是需要原⽂引⽤的条款解释题，再回退到原始⽂本检索
知识图谱的代价是预处理阶段需要消耗额外 Token 做信息抽取，但在答题阶段可以⼤
幅减少重复阅读，整体 Token 效率可能更优。⽽且如果允许的话，离线抽取不受 500
万 Token 总预算限制，性价⽐很⾼。不过赛题⽅是有对知识图谱构建的过程计算对应
的token的，是否能满⾜我们的实际要求的话，还需要使⽤实战来进⾏测试了。
