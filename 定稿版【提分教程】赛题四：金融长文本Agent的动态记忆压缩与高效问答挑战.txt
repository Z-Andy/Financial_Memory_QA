1. 赛题题⽬类型分析
A 组 100 道题看似都是"读⽂档选答案"，实际按解题⽅法可以分成⼏类。理解这些类
型，才能明⽩为什么"搜索到⽚段直接回答"远远不够。
1.1 公式计算题
题⽬给出具体数值，要求从条款中提取计算公式后代⼊计算。
示例：ins_a_002（退保⾦排序）
平安智盈⾦⽣累计所交保费 10 万元，保单账户累计收益 2 万元，在第 7 个保单年度
末退保；国寿增益宝个⼈账户价值 12 万元，在第 7 个保单年度末退保（退保费⽤
0%）；平安富鸿⾦⽣现⾦价值 9 万元。则三者退保所得⾦额从⾼到低排序为？
这道题需要从 3 份保险条款中分别找到退保⾦的计算公式：
• 平安智盈⾦⽣：第 6-10 年现⾦价值 = 累计保费 + 收益 × 75% → 10 + 2 × 0.75 =
11.5 万
• 国寿增益宝：退保费⽤ 0%，退保⾦ = 个⼈账户价值 = 12 万
• 平安富鸿⾦⽣：现⾦价值直接给出 = 9 万
排序：12 > 11.5 > 9 → 选项 A 正确。
这类题的难点是公式分散在不同⽂档，必须逐份定位条款，提取参数，再做算术。只
搜"退保"两个字得到的⽚段⼤概率漏掉"第 6-10 年"这个关键条件。
示例：ins_a_003（多保险联合赔付计算）
李某因⽩⾎病复发住院，总费⽤ 8 万元，医保报销 3 万元，⾃费 5 万元。众安⽩⾎病
医疗险免赔额为 0；平安 e ⽣保（计划⼆，免赔额 5000 元）；太保团体百万医疗（免
赔额 1 万元）。三家保险公司共应赔付多少？
这道题涉及 3 份医疗险条款，每份的赔付规则不同：有的先扣免赔额再按⽐例赔，有
的 0 免赔全额赔，还要看费⽤是否属于保险责任。最终答案需要通过多条款交叉计算
得出，不是单⽂档搜索能解决的。

1.2 跨⽂档对⽐题
题⽬明确要求对⽐两份或多份⽂档中的数据或条款。
示例：fin_a_001（⽐亚迪连续两年年报对⽐）
根据⽐亚迪连续两年的年度报告，下列关于公司经营业绩变化的描述中，哪些是准确
的？
A. 2025 年营业收⼊较 2024 年实现增⻓
B. 2025 年归属于上市公司股东的净利润出现下滑
...
需要分别从 2024 年报和 2025 年报中提取营业收⼊、净利润、经营现⾦流、研发投⼊
等数据，再逐项对⽐增⻓/下滑关系。单看任意⼀份年报都不够。
示例：fin_a_005（跨公司财务指标对⽐）
基于⽐亚迪与美的集团 2024 年年度报告的数据对⽐，关于两家企业关键财务指标的
结论，哪些是成⽴的？
这道题需要同时打开⽐亚迪和美的两份年报，提取同⼀类指标（营业收⼊规模、研发
投⼊强度、经营现⾦流等）做横向⽐较。如果检索模块只返回了⽐亚迪的数据，美的
的数据没召回，答案就直接错⼀半。
1.3 多⽂档联合推理题
题⽬涉及多个产品/法规/条款，需要把各⽂档的信息拼起来做综合判断。
示例：ins_a_001（四产品⾝故保险⾦排序）
关于四个养⽼保险/寿险产品的"⾝故保险⾦"，以下计算结果正确的是？假设所有产品
的已交保费均为 100 万元...平安智盈⾦⽣（领取⽇前）保单账户价值 90 万元；国寿增
益宝（⼀⼈，40 岁）基本保额 90 万元，个⼈账户价值 85 万元...
涉及 4 份保险条款，每份的⾝故保险⾦计算公式都不⼀样：有的是"已交保费 vs 现⾦
价值取⼤"，有的是"已交保费 × ⽐例 + 账户价值"，有的是"基本保额 × 系数"。必须四
份条款都读到、都算对，才能排序。
示例：ins_a_005（多产品免责范围判断）
下列哪些情形属于相应保险产品的免责范围？
A. 平安智盈⾦⽣：被保险⼈因酒后驾驶导致失能失智护理状态
B. 平安安佑福重疾险：被保险⼈因感染艾滋病病毒（⾮输⾎、职业、器官移植）导致
重⼤疾病...
4 个选项分别来⾃ 4 份不同条款，每个选项都要到对应条款⾥找"责任免除"章节，确认
该情形是否在免责列表中。这本质上是4 次独⽴的条款定位任务，拼在⼀起成了⼀道
多选题。

1.4 条款定位题
题⽬问的是某⼀条款的具体规定，答案就在⽂档的某个章节⾥。
示例：ins_a_004（等待期内出险处理）
关于"等待期"内出险的处理，以下哪种情形下保险公司需要承担赔偿责任？
A. 投保平安安佑福重疾险第 50 天，因意外⻋祸导致双⽬失明（符合重⼤疾病）
...
4 个选项涉及 4 份不同保险，每份的"等待期"条款可能都有例外规定（⽐如"因意外伤
害发⽣上述情形的，⽆等待期"）。需要逐份定位 等待期 条款，再 判断该 选项 情形 是 否
适⽤例外。
1.5 事实查找题
题⽬问的是⽂档中直接记载的事实，不需要计算，只需要定位。
示例：res_a_001（⾏业数据查找）
关于银保渠道发展历程及银⾏ IT 市场规模的描述，下列哪些选项是正确的？
A. 韩国寿险银保渠道保费贡献率在 2022 年达到了 56%
B. 2005 年⾄ 2018 年期间，银保渠道实现复合增速 9.9%
...
这类题在研报中通常有明确的数字和结论，但研报动辄⼏万字，数字散落在不同段落。
如果检索只返回了前半部分，可能恰好漏掉了"2022 年"这个限定词。
⼩结
这五种类型不是互斥的，很多题同时涉及多种能⼒。⽐如 ins_a_003 既是多⽂档联合
推理（3 份条款），⼜是公式计算（每份条款的赔付公式不同）。fin_a_001 既是跨⽂档
对⽐（两年年报），⼜是事实查找（具体数值）。
这意味着⼀个有效的系统⾄少要解决两个问题：
1. 如何确保所有相关⽂档都被读到（跨⽂档题不能只读⼀篇）
2. 如何在⽂档中精确定位关键信息（计算题要找到公式，条款题要找到对应章节）
硬截断策略对第 1 个问题⽆能为⼒（4 篇⽂档每篇截断 4000 字，总⻓度可能超限），
对第 2 个问题取决于关键信息是否恰好在前 4000 字内。后续优化⽅向，本质上都是
在解决这两个问题。

1.6 什么是 RAG（检索增强⽣成）
上⾯两个问题的标准解决⽅案，在 NLP 领域有⼀个专⻔的名字：RAG（Retrieval-
Augmented Generation，检索增强⽣成）。它的核⼼思想很简单：在让模型回答之前，
先从⽂档中把相关内容找出来，只把最相关的⽚段交给模型，⽽不是把整篇⽂档都塞
过去。
标准的 RAG 流程分三步：
（1）索引（Indexing）
把所有⽂档切成⼩块（chunk），每 块 ⼏百到⼏ 千 字 符 ， E然mb后ed⽤ding 模型把每
块⽂本编码成向量，存⼊向量数据库。这样每段⽂本都有了⼀个"地址"，可以通过向量
相似度快速定位。
（2）检索（Retrieval）
收到⽤户问题后，把问题也编码成向量，到向量数据库⾥搜索最相似的 top K 个⽂本
块。这⼀步的本质是"找证据"，即从海量⽂档中挑出可能包含答案的那⼏段。
（3）⽣成（Generation）
把检索到的⽂本块和原始问题⼀起拼成 prompt，交给 LLM ⽣成最终答案。因为 LLM
看到的不是整篇⽂档，⽽是经过筛选的最相关⽚段，所以既减少了噪⾳，⼜降低了
Token 消耗。
RAG 与本赛题的关系
⾦融⻓⽂本问答本质上就是⼀个 RAG 任务：给定⼀个问题（如"平安智盈⾦⽣的退保
⾦怎么算"），系统需要从 指 定⽂档检中索到相关条款，然后⽣成正确答案。本⽅案的
硬截断等于跳过了"检索"这⼀步，直接把⽂档前 4000 字全部塞给模型，让模型⾃⼰从
噪⾳⾥找信号。后续优化的核⼼，就是逐步补⻬ RAG 的各个环节：更好的分块策略、
更准的检索算法、更合理的⽚段组装⽅式。

2. PDF 解析⽅案
RAG 系统的质量上限由两个因素决定：检索策略和⽂档解析质量。Baseline 教程第七
章已经指出了 PDF 解析质量是第⼀个瓶颈，但只给了⽅向没有展开。这⼀章详细对⽐
主流⽅案，并解释 baseline 的选择逻辑。
2.1 为什么 PDF 解析是第⼀步且最重要
整个 pipeline 的链路是：PDF → 解析 → ⽂本 → 切⽚ → 检索 → LLM 回答。如果第
⼀步就把表格变成了乱码、公式丢失了、段落顺序乱了，后⾯所有环节都在错误的⽂
本上⼯作。这种错误⽆法通过优化检索或 prompt 来补救。
保险条款和财报类 PDF 有⼏个让解析器头疼的特点：
• 复杂表格：保费表、现⾦价值表、赔付⽐例表，⾏列密集且嵌套层级深
• 数学公式：精算公式、财务⽐率公式，上下标和分式结构容易丢失
• 双栏/多栏布局：部分研报和合同使⽤多栏排版，阅读顺序容易错
• ⻚眉⻚脚混⼊正⽂：⻚码、章节标题、公司名称等重复内容污染正⽂
• 扫描件 vs 电⼦件：部分⽂档是扫描图⽚（需要 OCR）， 部 分是 原 PD⽣F（可直
接提取⽂字），两者需要不同的处理策略
2.2 主流⽅案对⽐
（1）轻量级：PyMuPDF4LLM（当前 baseline ⽅案）
PyMuPDF4LLM 是基于 PyMuPDF（原 fitz）的轻量封装，专⻔为 LLM 场景设计。它
的核⼼能⼒是把 PDF 直接转为 Markdown ⽂本，保留标题层级（通过字号和字体推
断），提取表格为 Markdown table 格式。
优势：
• 纯 Python 库，安装简单（pip install pymupdf4llm）
• 不需要 GPU，纯 CPU 运⾏
• 速度快，⼀份 50 ⻚ PDF 约 5-10 秒完成
• 对原⽣ PDF（⾮扫描件）的⽂字提取质量尚可
• 输出 Markdown 格式，直接兼容下游处理
局限：
• 不包含 OCR 能⼒，扫描件 PDF ⽆法处理
• 表格提取基于规则，复杂嵌套表格容易错位

（2）中量级：Marker
Marker（VikParuchuri, 2024, 持续更新⾄ 2025 年 1.4.0 版本，GitHub:
VikParuchuri/marker）[1] 是⼀个基于 pipeline 架构的 PDF 转 Markdown ⼯具。它的

3. RAG 检索优化
Baseline 的硬截断⽅案本质上跳过了检索，直接把⽂档前半部分塞给模型。要突破 17%
的准确率天花板，第⼀步就是引⼊检索模块，但必须注意检索引⼊的连贯性断裂⻛险。
3.1 分块策略
⽂档切⽚是 RAG 系统的地基。切得不好，后⾯的检索和⽣成都受影响。
固定⼤⼩分块：按固定字符数或 token 数切分，实现简单但容易切断语义。⽐如"因意
外伤害发⽣上述情形的，⽆等待期"这句话如果被切成两块，前半句在⼀块的末尾、后
半句在另⼀块的开头，检索可能只命中其中⼀块，模型看到的就不完整。
滑动窗⼝分块：相邻块之间有重叠（通常 10%-20%）， 减 少 边界 信息 丢 失 。 适合 保险
条款这种"前后⽂紧密关联"的⽂档。重叠太多会导致检索结果重复，占⽤ token 预算。
语义分块：按段落、章节等⾃然边界切分，保持每个块内的语义完整。Markdown ⽂
档天然适合这种策略（按标题层级切分）。缺点是块⼤⼩不均匀，有的块可能过⻓。
多粒度分块（Small-to-Big / Parent-Child）：这是更进阶的策略。把⽂档切成两层：
• ⼦块（Child Chunks）：较⼩（200-500 token），⽤ 于 向 量检 索。 ⼩块 嵌 ⼊ 更 聚焦 ，
检索精度更⾼。
• ⽗块（Parent Chunks）：较⼤（800-2000 token）， 包含⼦ 块 的 完整 上下 ⽂。 检
索命中⼦块后，把对应的⽗块也⼀并取出来送给 LLM。
这种做法的好处是"检索⽤⼩块（精确）、回答⽤⼤块（完整）"。实验表明，⼦块检索
+ ⽗块上下⽂的⽅案在准确率上⽐单⼀⼤块⽅案⾼ 10%-15%。
推荐起步⽅案：按 Markdown 的 ## 标题切分 + 滑动窗⼝，块⼤⼩控制在 800-1200
token，重叠 15%。有精⼒的话，升级到多粒度分块（⼦块 400 token 检索 + ⽗块
1200 token 回答）。
3.2 混合检索
单⼀检索策略各有盲区：
• BM25 对精确匹配敏感（数字、条款编号），但⽆法理解"退保费⽤"和"退保⼿续费"
是同⼀个东⻄
• 向量检索（Embedding）能捕捉语义相似性，但对数字和年份不够敏感
混合检索 = BM25 + 向量检索，两路同时召回，结果融合后⼀起送进 LLM。融合⽅式：
• 简单合并去重：两路结果拼接，去重后按原始⽂档顺序排列
• RRF（Reciprocal Rank Fusion）：按每路结果的排名位置加权融合，不依赖绝

视觉检索：跳过 OCR 的另⼀种思路
前⾯的所有⽅案都遵循同⼀个范式：PDF → 解析为⽂本 → 切⽚ → ⽂本检索。但如果
PDF 解析本⾝就是瓶颈（表格乱码、公式丢失、双栏错序），有没有可能直接跳过解
析？
ColPali（2024）和 ColQwen（2025）[15] 提出了视觉检索的思路：把 PDF ⻚⾯渲
染成图像，⽤视觉语⾔模型（VLM）直接从⻚⾯图像中提取 patch 级别的嵌⼊，然后
基于这些视觉嵌⼊做检索。整个过程不需要 OCR、不需要版⾯分析、不需要
Markdown 转换。
具体做法：
1. 把每⻚ PDF 渲染为⾼分辨率图像（如 448×448）
2. VLM 对图像做 patch 编码（如 ColPali 产出每⻚ 1024 个 patch embedding）
3. 查询时，问题⽂本和⻚⾯图像的 patch embedding 做 MaxSim 匹配（取每个
query token 与所有 patch 的最⼤相似度之和）
4. 返回最相关的⻚⾯
优势：
• 完全绕过了 PDF 解析的所有问题：表格、公式、双栏都不需要处理
• 布局信息天然保留（表格的视觉结构、公式的排版位置）
• ViDoRe 评测基准上，ColPali 的检索精度超越了传统 OCR + ⽂本检索⽅案
局限：
• 需要 VLM（3B+ 参数），GPU 必须
• 返回粒度是整⻚，不是精确的⽂本⽚段：模型需要⾃⼰从⻚⾯图像中"读"出答案
• ⽬前以英⽂⽂档为主，中⽂⽂档的视觉检索效果待验证
• ColModernVBERT 提供了⼀个 250M 参数的轻量替代，精度接近 ColPali 但模型
⼩ 10 倍
对本赛题的启示： 视觉检索提供了⼀个激进的思路：如果 MinerU 或 Marker 的解析效
果在特定⽂档上仍然不理想，可以直接把⻚⾯当图⽚检索，搭配多模态 LLM（如
GPT-4o、Qwen-VL）来"看图答题"。这在保险条款的复杂表格和公式场景下可能有奇
效。但 GPU 要求和中⽂适配是两个现实障碍，建议作为实验性⽅向⽽⾮主⽅案。

3.6 前沿 RAG 范式

是调参数，⽽是改变了检索的底层结构：从"搜碎⽚"升级到"理解结构"。
理解这些范式的关键，在于搞清楚它们各⾃在解决什么问题。传统 RAG 有三个根本缺

1. 扁平化：所有⽂档块在同⼀层，检索只看块级别的相似度，⽆法理解跨块的语义

2. 局部性：擅⻓回答"这段话说了什么"，但回答不了"这些⽂档整体讲了什么"（四份

下⾯的范式各⾃针对其中⼀个或多个缺陷。

GraphRAG：⽤知识图谱做检索
GraphRAG（Microsoft Research, 2024 年 4 ⽉发布, arXiv:2404.16130）[5] 的核⼼思
路：不要只把⽂档切成碎⽚去搜，⽽是先从⽂档中提取实体和关系，构建知识图谱，
再基于图谱做检索和推理。
传统 RAG 本质上在做"向量空间中的邻居搜索"：把问题和⽂档块都映射到向量空间，
找距离最近的块。这种做法的上限是"找相似的⽂本⽚段"，但⽆法回答需要跨⽚段推理
的问题。⽐如"这四份保险条款中，哪份对意外⾝故的赔付最⾼？"，这个问题需要先分
别理解四份条款的⾝故保险⾦计算规则，再横向⽐较，⽽不是找到某个"相似"的⽚段就
够了。
GraphRAG 的完整流程：
1. 实体与关系抽取：⽤ LLM 从源⽂档中提取实体和关系。实体可以是产品名（"平
安智盈⾦⽣"）、概 念 "（现⾦价值"）、数值（"75%"）、条款 编 号"第（ 7 保单年度"）；关
系描述实体之间的逻辑连接（"现⾦价值 → 计算公式 → 累计保费 × 75%"、"意外⾝故
→ 适⽤规则 → ⽆等待期"）。
2. 知识图谱构建：将抽取出的实体作为节点、关系作为边，构建⼀张有向/⽆向图。
这张图不是简单的关键词共现⽹络，⽽是携带了语义信息的知识结构。
3. 社区检测：⽤ Leiden 算法在图结构上发现紧密关联的实体群组（社区）。每个社
区对应⼀个主题领域，⽐如"退保规则社区"可能包含"退保费⽤""退保费⽤率""现⾦价值
""保单年度"等实体及其关系。
4. 社区摘要⽣成：对每个检测出的社区，⽤ LLM ⽣成⼀个总结性的摘要，描述这个
社区的核⼼信息。这⼀步是 GraphRAG 区别于传统图⽅法的关键：不是把原始⽂本⽚
段直接存到图⾥，⽽是对每个主题⽣成⾼层摘要，使得检索时可以直接读取"关于退保
规则的核⼼要点是什么"。
5. 两种检索模式：
￮ Local Search（局部搜索）：查询时匹配相关实体，从实体出发沿图的边扩散
到邻居节点，获取与问题直接相关的实体和关系。适合事实查找类问题（"平安智
盈⾦⽣的现⾦价值怎么算？"）。
￮ Global Search（全局搜索）：查询时匹配相关社区，⽤社区的摘要⽣成部分
回答，再将所有部分回答汇总成最终答案。适合全局理解类问题（"四份条款的退
保规则有什么系统性差异？"）。这⼀步本 质上 是 Map-Reduce：社区摘要做 Map，
LLM 汇总做 Reduce。
对本赛题的适⽤性：
跨⽂档对⽐题和多⽂档联合推理题是 GraphRAG 的最优场景。⽐如 ins_a_001（四产
品⾝故保险⾦排序），传统 RAG 需要逐份⽂档检索每份产品的⾝故保险⾦规则，再在

prompt ⾥⼿动对⽐。GraphRAG 可以在知识图谱中直接查询"⾝故保险⾦"这个实体，

LightRAG：轻量级图 RAG
LightRAG（⾹港⼤学 HKUDS 实验室, 2024 年 10 ⽉发布, EMNLP 2025,
arXiv:2410.05779）[6] 是 GraphRAG 的轻量级替代⽅案，保留了图结构的优势，但⼤
幅降低了构建成本和检索延迟。
与 Microsoft GraphRAG 的关键差异：
GraphRAG 的社区检测 + 逐社区摘要是⼀个重操作：Leiden 算法需要遍历整张图，每
个社区的摘要⼜需要独⽴的 LLM 调⽤。LightRAG 不依赖社区检测，⽽是⽤双向量索
引来同时⽀持局部和全局检索：
1. 实体向量索引：每个实体节点有独⽴的向量表示（由实体名称和描述的
embedding 组成）。查询时，问题向量与实体向量做相似度匹配，找到相关实体后沿关
系边扩散到邻居节点。这对应 GraphRAG 的 Local Search。
2. 关系向量索引：每条关系边也有独⽴的向量表示（由关系描述和两端实体信息的
embedding 组成）。查询时，问题向量与关系向量做相似度匹配，直接找到相关的关系。
这能捕捉到跨实体的语义模式（⽐如"退保费⽤率逐年递减"这种趋势性关系），⽽不仅
仅是单个实体的匹配。
3. 关键词引导查询：LightRAG 从⽤户问题中提取关键词，将这些关键词的向量拼接
成组合查询向量。这种⽅式⽐直接⽤整个问题做向量检索更聚焦，减少了语义稀释。
增量更新能⼒：
LightRAG 的图结构⽀持增量插⼊——新⽂档加⼊时，只需要提取新⽂档的实体和关系、
计算它们的向量、插⼊图结构中，不需要重建已有索引。这对本赛题可能很重要：A
榜只有 100 题对应的⼀批⽂档，如果 B 榜下发新⽂档，LightRAG 可以直接增量更新，
⽽ GraphRAG 需要重新跑社区检测。
对本赛题的定位：
LightRAG 适合作为从标准 RAG 向图 RAG 过渡的第⼀个实验⽅案。它的构建成本接
近传统 RAG（主要开销在实体抽取，不需要社区检测），但能同时⽀持局部事实查找
和全局对⽐推理。如果本赛题后续出现需要跨⽂档联合推理的难题，LightRAG 的性价

RAPTOR：树状递归摘要检索
RAPTOR（Recursive Abstractive Processing for Tree-Organized Retrieval）
（2024 年 1 ⽉发布, arXiv:2401.18059）[8] 解决的是"扁平检索丢失全局上下⽂"的问
题，思路和 GraphRAG 不同：GraphRAG ⽤图结构，RAPTOR ⽤树结构。
为什么需要树结构：
传统 RAG 把所有⽂档块平铺在⼀层。检索时只看单个块的相似度，但有些问题的答案
跨越了多个块。⼀个典型的保险条款场景：某产品的退保规则可能分布在"第⼆章 现⾦
价值"（定义公式）、"第五章 退保"（退保流程）、"附录"（费率表）三个不同章节。这
三个章节在⽂档中相距很远，但在语义上紧密关联。扁平检索很难同时召回这三个章
节的⽚段。
RAPTOR 的构建流程：
1. 叶⼦层：将⽂档切成基础块（如每⻚ Markdown），每个 块 做embedding
2. 聚类：⽤⾼斯混合模型（GMM）对叶⼦节点的 embedding 做软聚类，每个节点
可以属于多个聚类
3. 摘要：对每个聚类，⽤ LLM 对聚类内的所有⽂本块⽣成⼀个摘要。这个摘要成为
上⼀层的⼀个节点
4. 递归：对上层节点重复聚类 + 摘要，直到只剩⼀个根节点
5. 最终结构：⼀棵多层树，底层是原始⽂本块（精确细节），中层是主题摘要（如"
退保规则""⾝故保险⾦规则"）， 顶 层 是 全 局 摘"该要产（品如的核⼼条款概述"）
检索时的层级选择：
• 简单事实查找（"平安智盈⾦⽣第 7 年的现⾦价值是多少？"）→ 从叶⼦层检索，
直接拿精确数值
• 中等复杂度（"该产品的退保规则有哪些注意事项？"）→ 从中层摘要节点检索，
获取主题级别的完整信息
• 全局理解（"该产品的整体保障结构是怎样的？"）→ 从顶层摘要节点检索
变体与改进：
• adRAP（Adaptive RAPTOR）：解 决 RAPTOR 在动态数据集上的问题。原始
RAPTOR 每次新增⽂档都需要重建整棵树（因为聚类会变）。adRAP 通过增量调整聚
类结构来避免全量重建，适合⽂档持续增加的场景。
• postQFRAP：不改变检索端的索引结构，⽽是在检索结果回来之后，对召回的⽚
段做⼀次"查询聚焦的递归摘要处理"：把检索结果按 RAPTOR 的思路聚类+摘要，⽣
成⼀个结构化的上下⽂再送给 LLM。优势是可以作为后处理模块接⼊任何检索管线。
对本赛题的适⽤场景：

⾏业研报类题⽬（如 res_a_001 银保渠道发展历程）和跨年度对⽐题（如 fin_a_001

Self-RAG 与 CRAG：让模型反思检索质量
传统 RAG ⽆条件信任检索结果：搜到什么就⽤什么，不管质量如何。这在实践中是个
严重问题：BM25 可能因为关键词匹配把不相关的⽚段排到前⾯，向量检索可能因为
语义模糊召回弱相关的内容。Self-RAG 和 CRAG 在这⾥加了"⾃我评估"机制。
Self-RAG（2024 年 10 ⽉发布, arXiv:2310.11511）[9] 的核⼼是"检索 + 反思 + 重试"
循环：
1. 按需检索：不是每个问题都需要检索。Self-RAG 先判断当前问题是否需要检索
（简单常识问题可能不需要）
2. 检索⽂档⽚段
3. 对每个⽚段做三⽅⾯评估：相关性（这个⽚段和问题相关吗？）、⽀持度（这个⽚
段⽀持还是反对某个观点？）、有⽤性（这个⽚段对回答有帮助吗？）
4. 对低质量⽚段重新检索或修正
5. ⽣成最终答案时，标注每个论断的来源⽚段和可信度
CRAG（Corrective RAG）（2024 年 1 ⽉发布, arXiv:2401.15884）[10] 在 Self-RAG
基础上更进⼀步，加⼊了⾃动纠错机制：
1. 检索⽂档⽚段
2. ⽤检索质量评估器对整体检索结果打分（0-1 分）
3. 根据分数分三级处理：
￮ 分数 ≥ 0.7（CORRECT）： 检 索 质量好 ，直接 使 ⽤
￮ 分数 0.3-0.7（AMBIGUOUS）： 质量 ⼀ 般 ，保 留当 前结果 同 时 ⽤不同关键 词
补充检索，两轮结果合并
￮ 分数 ≤ 0.3（INCORRECT）： 质量 差 ， 丢 弃 检 索 结果 ， 切 换 到 全 ⽂阅读 或 其
他策略
4. 最终⽣成前再做⼀次"幻觉检查"，过滤掉检索结果中不存在的论断
CRAG 的关键洞察： 检索质量的评估不需要复杂的模型。⽤检索结果中每个⽚段与问
题的关键词重叠度、⽚段间的语义⼀致性等轻量特征，就能⼤致判断检索质量。只有
当评分落在中间区域时才需要 LLM 介⼊做进⼀步判断。

• CORRECT → 直接⽤检索结果⽣成答案
• AMBIGUOUS → 换关键词重新检索同⼀⽂档集，或者扩⼤检索范围（如从只搜前

HyDE：⽤假设答案做检索
HyDE（Hypothetical Document Embeddings）（2023 年 12 ⽉发布,
arXiv:2212.10496）[11] 解决的是"⽤户问题⽤词和⽂档⽤词不⼀致"导致检索失败的问
题。
问题场景： ⽤户问"怎么退保最划算"，⽂档⾥写的是"现⾦价值计算⽅式"和"退保费⽤
率表"。问题中的"划算"是⼝语化表达，⽂档中的"现⾦价值"是专业术语，两者在向量
空间中距离很远，直接向量检索可能召回到完全不相关的内容。
HyDE 的做法： 先让 LLM 凭空⽣成⼀份假想的"理想答案"，再把这份假想答案（⽽⾮
原始问题）转成向量去检索。假想答案是⽤⽂档⻛格写的（包含"现⾦价值""退保费⽤
""已交保费"等专业术语），所以更容易匹配到真实的相关⽚段。
流程：⽤户问题 → LLM ⽣成假设答案 → 假设答案 Embedding → 向量检索 → 返回
真实⽚段。
代价是每次检索多⼀次 LLM 调⽤（⽣成假设答案），但检索命中率提升显著：在多个
benchmark 上，HyDE 的召回率⽐直接向量检索⾼ 10%-20%。对本赛题的⼝语化问题
（如"哪个产品最划算"）特别有效。
HyDE + 混合检索的实践建议： 根据"Searching for Best Practices in RAG"（2024 年
7 ⽉发布, arXiv:2407.01219）[12] 的系统评测，HyDE + 混合检索（BM25 + 向量）
是⽬前综合表现最好的默认配置。该评测在多个 benchmark 上对⽐了不同检索策略，
HyDE + Hybrid Search 的检索质量显著优于单独的查询改写（Query Rewriting）或查
询分解（Query Decomposition）。

多跳推理 RAG（Multi-hop RAG）
多跳推理是指答案不能从单个⽂档⽚段直接获得，需要通过多个⽚段的推理链来推导。
本赛题中这类问题很常⻅："某产品在第 N 年的退保⾦额是多少？"需要先找到退保公
式（第⼀跳），再找到该年度的退保费⽤率（第⼆跳），再找到该年度的现⾦价值（第
三跳），最后代⼊计算。
传统 RAG 对多跳问题的处理很弱：检索可能只召回第⼀跳的⽚段（退保公式），但漏
掉了后两跳（费⽤率和现⾦价值）。针对多跳 RAG 的改进⽅案：
• Iterative RAG（迭代检索）：第⼀轮检索 → ⽣成部分答案 → 根据部分答案⽣成
新查询 → 第⼆轮检索 → 汇总。每轮检索聚焦⼀个⼦问题
• IRCoT（Interleaving Retrieval with Chain-of-Thought）：把思维链推理和检索
交替进⾏。每推⼀步就检索⼀次，⽤检索结果来验证和修正推理⽅向
• Beam Search RAG：对多跳路径做束搜索，同时探索多条推理链，最后选证据最
充分的⼀条
Agentic RAG：让 Agent ⾃主编排检索策略
前⾯所有的 RAG 范式都预设了检索策略：什么时候搜、搜什么、搜⼏次，都是提前写
死的。Agentic RAG 的思路是把这个决策权交给 LLM ⾃⼰：通过 Function Calling，
LLM ⾃主决定：
• 当前问题需要检索吗？（简单常识题可能不需要）
• 搜什么关键词？（可能需要多轮尝试不同关键词）

4. 上下⽂管理与动态记忆压缩
赛题名称⾥就有"动态记忆压缩"，这其实定义了本赛题的核⼼技术挑战。前⾯三章
（PDF 解析、RAG 检索）解决的是"怎么找到有⽤信息"，这⼀章解决的是"怎么在有
限 Token 预算内把最重要的信息喂给模型"。
4.1 问题的本质：上下⽂窗⼝是稀缺资源
每道题的推理链路可以拆成：读⽂档 → 提取信息 → 判断 → 输出答案。每⼀步都消耗
Token，但预算有限（赛⽅总预算 500 万 Token）。这 意味着 不能 把 所有⽂档 全 读⼀ 遍 ，
也不能把检索结果全塞进 prompt。
上下⽂管理的核⼼问题可以表述为：给定 Token 预算 B，如何从候选信息池中选择⼀
个⼦集，使得模型回答正确的概率最⼤化？
候选信息池包括：
• ⽂档原⽂⽚段（检索结果）
• 上⼀步推理的中间结果
• 题⽬本⾝和选项
• Prompt 模板
• 验证/⾃检的⼆次推理结果
4.2 动态截断
Baseline 的硬截断固定取前 4000 字符。更好的策略是让模型⾃⼰决定什么时候停⽌
阅读。
实现思路：分段读取⽂档，每读⼀段就让模型判断"当前信息是否⾜够回答问题"。如果
⾜够就停⽌，不够就继续读下⼀段。这本质上是 Agent 化的第⼀步。
更精细的做法是按题型设定不同的阅读预算。事实查找题可能读前 2000 字符就能找
到答案；跨⽂档对⽐题需要读多份⽂档，每份可能 1000 字符就够；公式计算题只需
要定位到公式所在的段落，读到公式后就可以停。
4.3 信息密度控制
保险条款和监管法规中有⼤量模板化⽂本（如"本合同的构成""投保须知"等），这些内
容对答题⼏乎没有价值，但占⽤了⼤量 token。
可以预处理时过滤掉⾼频⽆意义段落，或者在检索时对"释义""名词定义"等章节降权。
更系统化的做法是按题型定义信息价值函数：对计算题，公式段落的权重最⾼；对条
款定位题，例外条件段落的权重最⾼。

4.4 动态记忆压缩：分层存储与选择性遗忘
这才是本赛题"动态记忆压缩"的核⼼。前⾯的截断和密度控制都是静态策略，动态压缩
需要在推理过程中持续做决策：什么该记住、什么该压缩、什么该丢弃。
⼯业界的实践：Agent 的记忆分层

Kimi Code CLI 的 Context Compaction [19]

1. Current Task State（当前任务状态：正在做什么）
2. Errors & Solutions（遇到的错误和解决⽅案）
3. Code Evolution（代码最终版本，丢弃中间尝试）
4. System Context（项⽬结构、环境配置）
5. Design Decisions（架构决策及理由）
6. TODO Items（未完成的任务）
Compaction 前后有 PreCompact / PostCompact hooks 可插拔，允许⽤户⾃定义压缩
策略（如阻⽌特定信息的压缩）。压缩后的历史替换原始对话，模型只看到摘要。
这个机制对本赛题的启发：压缩的核⼼是有选择地保留、改写、丢弃。Kimi Code 的 6

1. 题⽬的完整⽂本和所有选项（最⾼优先级，绝不压缩）
2. 从⽂档中提取出的公式、数值、条件判断（核⼼证据）
3. 检索到的原⽂⽚段中与题⽬直接相关的部分
4. ⽂档章节标题和结构信息（⽤于定位和引⽤）
5. 中间推理过程的摘要
6. 检索到的原⽂中经判断为不相关的部分（最低优先级，优先丢弃）

Hermes 的 Trajectory Compressor [20]
Hermes 是⾯向通⽤ Agent 任务的框架，其 trajectory_compressor.py 实现了⼀套
完整的对话历史压缩管线，虽然是为训练数据压缩设计的，但核⼼算法可以直接迁移
到推理场景。
压缩算法的完整流程：
第⼀步：计算压缩需求。 遍历 trajectory 中每个 turn，⽤ tokenizer 计数。如果总
tokens 未超过⽬标上限（默认 15250），直接 跳 过 压缩 。
第⼆步：确定受保护区域。 在 trajectory 中标记不可压缩的 turns：
• 头部保护：第⼀个 system 消息、第⼀个 human 消息、第⼀个 gpt 回复、第⼀个
tool 调⽤。这些是任务的初始上下⽂，丢弃后模型会丢失任务定义。
• 尾部保护：最后 N 个 turns（默认 N=4）。这些 包含 最 新 的 推 理 状态和 即将 做出的
决策。
第三步：计算压缩区域。 受保护头部之后、受保护尾部之前的所有 turns 构成"可压缩
区域"。从这个区域的起始位置开始，逐 turn 累积 tokens，直到累积量满⾜：
Plaintext
累计 tokens ≥ (总 tokens - 目标上限) + 摘要目标 tokens
如果遍历完整个可压缩区域仍不满⾜，则压缩整个区域。
第四步：LLM ⽣成摘要。 将被压缩的 turns 拼接为⼀段⽂本，发送给单独的摘要模型
（默认 gemini-3-flash，temperature=0.3）。 摘 要prompt 明确要求包含：
• 助⼿执⾏了哪些操作（⼯具调⽤、搜索、⽂件操作）
• 获得了什么关键信息或结果
• 做出了什么重要决策或发现
• 相关的数据、⽂件名、数值、输出
摘要以 [CONTEXT SUMMARY]: 为前缀，⽬标⻓度 750 tokens。
第五步：组装压缩后的 trajectory。 头部 turns（原⽂保留）+ 摘要消息（作为
human 消息插⼊）+ 尾部 turns（原⽂保留）。如果 system 消息中配置了
add_summary_notice，会在 system prompt 末尾追加提示⽂字，告知模型部分历史
已被压缩。
压缩效果度量： 每条 trajectory 压缩后记录详细 metrics：
• compression_ratio：压缩后 tokens / 原始 tokens
• tokens_saved：节省的 tokens 数

• turns_removed：被移除的 turns 数
• still_over_limit：压缩后是否仍超预算
对本赛题的关键启发：

• "⾸"是什么？ 题⽬⽂本 + 选项（不可压缩，必须完整保留）
• "尾"是什么？ 最终推理步骤（列出公式 → 代⼊数值 → 计算结果 → 对⽐选项 →

• 摘要替代：⽤ LLM（或同⼀个⼩模型）⽣成⽂档阅读摘要，替代原始阅读内容。

Claude Code 的四层渐进压缩 [21]
Claude Code 把压缩拆成四层，从便宜到贵逐级尝试：
Plaintext
Layer 1: Snip（裁剪工具输出）→ 免费，不调 LLM
Layer 2: Microcompact（基于 prompt cache 的轻量压缩）

Layer 4: Autocompact（LLM 全量总结）→ 最贵
核⼼设计原则：只有低层搞不定才升级。Snip 只是把⽼⼯具结果截断为头尾各保留⼀

Snip 就够了，只有 20% 需要 LLM 介⼊，整体 token 效率远⾼于每次都调 LLM。
对本赛题的启发：检索返回的⽂档⽚段如果太⻓，先 Snip（只保留包含关键词的段落
+ 前后各⼀段上下⽂）；Snip 后仍然超预算再调 LLM 做摘要。不要⼀上来就让模型总

Codex 的 compaction 有⼀个关键设计：压缩后的历史不是纯摘要，⽽是
initial_context + 精选 user 消息 + summary + 最近 N 条。原因很简单，
LLM ⽣成的 summary 会丢失精确信息。

四层架构的完整定义：
L1：全局内存索引（global_mem_insight.txt）
硬约束 ≤30 ⾏、期望 <1k tokens。内容只包含：

L2：全局事实库（global_mem.txt）

MemGPT（UC Berkeley, 2023）[16] 提出了⼀个根本性的视⻆转换：给 LLM ⼀个分

窗⼝本⾝。
MemGPT 的核⼼类⽐是操作系统的虚拟内存管理：
• 主上下⽂（Main Context）：LLM 的固定上下⽂窗⼝，类⽐物理 RAM。容量有限
（如 4K/8K/32K tokens）， 但 访 问速 度 最L快LM 。在这⾥做推理。
• 外部上下⽂（External Context）：⼏乎⽆限的⻓期存储，类⽐磁盘。存储完整的
对话历史、⽂档、⽤户信息等。访问速度慢（需要检索），但容量⼏乎不受限制。
MemGPT 的关键创新在于：LLM ⾃⼰控制"换⻚"过程。 传统做法是开发者写死截断
策略（取最后 N 轮对话），MemGPT 的做法是让 LLM 通过 Function Calling 主动管理
记忆：
• 当主上下⽂快满时，LLM 可以调⽤ conversation_search 或
archival_memory_search 来检索需要的信息
• LLM ⾃⼰判断当前上下⽂中哪些信息不再需要，将其"换出"到外部存储
• LLM ⾃⼰判断下⼀步推理需要哪些外部信息，将其"换⼊"主上下⽂
这本质上是把记忆管理从"系统层的固定规则"升级为"应⽤层的智能决策"。
MemGPT 的记忆类型：
1. 会话记忆（Conversational Memory）：对话历史的滚动窗⼝。超出窗⼝的历史被
⾃动分⻚存储，LLM 通过 conversation_search 函数按⽇期或关键词检索。
2. 档案记忆（Archival Memory）：⽂档、⽂章、知识库等外部资料。LLM 通过
archival_memory_search 函数做语义检索。这部分最接近本赛题的"⽂档检索"场景。
3. ⼯作记忆（Working Memory）：当前推理状态。LLM 可以通过
core_memory_append 和 core_memory_replace 函数主动更新（⽐如"⽤户刚才说了
喜欢 Python，我把这个记下来"）。

LLM 的计算短板
⼤语⾔模型本质上是"⽂字接⻰"引擎，不是计算器。它在做数学运算时其实是在"背诵"
训练数据⾥⻅过的计算结果，⽽不是真正执⾏算术。举例：
• 题⽬要求计算 (8亿 × 6.67% + 12 亿 × 75%) / (1 - 30%)
• LLM 直接推理：可能在某⼀步算错中间结果，或者搞混运算顺序
• 结果：选项 A 和选项 B 的数值差 0.3%，LLM 选错了
这不是模型不够聪明，⽽是概率⽣成天然不适合精确计算。研究（ARTIST, 2025）表
明，即使是最强的 LLM 在多步数值计算中的错误率也显著⾼于使⽤ Python 解释器辅

解决⽅案：让 LLM 写代码⽽不是算答案
核⼼思路：不让 LLM ⾃⼰算，⽽是让它写⼀段 Python 代码，⽤ Bash ⼯具执⾏代码，
再把执⾏结果还给 LLM 做最终判断。
这是终端 Agent（Terminal Agent / Coding Agent）模式在⾦融 QA 中的典型应⽤。参

1. 模型不直接输出答案，⽽是输出可执⾏的代码
2. 通过 Bash ⼯具执⾏代码，获取精确的数值结果

参考架构：ReAct 循环 + Bash ⼯具
ReAct（Reasoning + Acting, Yao et al., 2023）是⽬前最主流的 Agent 推理范式，也
是 Claude Code、OpenHands 等终端 Agent 的基础。
ReAct 的循环是：思考（Thought）→ ⾏动（Action）→ 观察（Observation）→
思考 → ...
在本赛题中：
Plaintext
Thought: "这道题需要计算三份保险的退保金。

Thought: "排序：120000 > 115000 > 90000，

这个循环的关键在于：Thought 和 Action 交替进⾏，模型每⼀步都基于上⼀步的观
察结果做决策，⽽不是⼀次性把所有动作都规划好再执⾏。

• 在 Function Calling 的 tools 定义中加⼊ execute_python(code: str) ⼯具
• ⼯具描述中明确说明：⽤于精确数值计算，不要⽤ LLM ⾃⼰做算术
• 模型输出代码后，在沙箱中执⾏（⽤ subprocess 或 exec）， 返 回stdout/stderr
• 代码执⾏超时设为 5 秒，防⽌死循环
• 如果代码报错，把错误信息返回给模型，让模型⾃⼰修
⼀个额外的收益：Bash 执⾏的 Token 消耗远低于让模型⼀步步推理。⼀段 Python 代

7.4 smolagents：⽤ Code Agent 范式重新组织⼯具调⽤
什么是 smolagents
smolagents（HuggingFace, 2025 年发布，持续更新中, GitHub:
huggingface/smolagents）[13] 是⼀个轻量级的 Agent 框架，GitHub 超过 24,000 星

这个区别看似只是"输出格式不同"，实际上影响了 Agent 的整个推理效率。
传统 Tool Calling vs Code Agent
传统 Function Calling 的流程：
Plaintext
模型推理 → 输出 JSON {"tool": "search", "params": {"query": "退保金
"}}
→ 外部系统解析 JSON → 执行工具 → 返回结果
→ 模型再推理 → 输出 JSON {"tool": "calculator", ...}
→ 外部系统再解析 → ...
每⼀步都需要 JSON 序列化/反序列化，且模型只能⼀次调⽤⼀个⼯具（或并⾏的⼏个

Code Agent 的流程：
模型推理后直接写⼀段 Python 代码：
Python
# 第一步：搜索退保条款

# 第二步：提取数值并计算
premium = 100000 # 从搜索结果中提取
interest = 20000
cash_value = premium + interest * 0.75
# 第三步：输出结果
final_answer(f"退保金为 {cash_value} 元")
代码⼀次性执⾏完毕，中间结果⾃动传递，不需要反复序列化。

效率提升
据 HuggingFace 的实验数据，Code Agent 相⽐ JSON Tool Calling 减少了约 30% 的

A.1 什么是 Agent（智能体）
Agent 是指⼤语⾔模型不再只做"你问我答"的被动响应，⽽是能够主动观察环境、做
出决策、调⽤⼯具、并根据反馈调整下⼀步⾏动的系统。
⽤⼀个⽐喻说明区别：
• 单次调⽤：你问"今天北京天⽓怎么样"，模型直接回答。这是"⼀问⼀答"，没有后
续动作。
• ⼯作流（Workflow）：你让模型"先查天⽓，再推荐穿⾐"，但每⼀步都是你提前写
死的，模型只是按顺序执⾏。⽐如第⼀步固定调⽤天⽓ API，第⼆步固定调⽤穿⾐推
荐模板。
• Agent：你给模型⼀个⽬标"帮我规划⼀次北京三⽇游"，模型⾃⼰决定：第⼀步查
天⽓，第⼆步搜景点，第三步查交通，第四步汇总成⾏程表。如果查到的景点因天⽓
原因不适合，模型会⾃⼰回到第⼀步重新查，⽽不是死板地继续执⾏原计划。
本赛题叫"⾦融⻓⽂本 Agent 的动态记忆压缩与⾼效问答挑战"，这⾥的 Agent 强调的
是模型在读取⻓⽂档的过程中，能够动态决定保留哪些信息、压缩哪些信息、什么时
候停⽌阅读。本 baseline 的硬截断⽅案其实不算真正的 Agent，因为它没有任何决策
能⼒，只是固定截断 4000 字符。
为什么赛题强调 Agent ⽽不是简单的问答
因为如果只是"把⽂档塞给模型、让模型选答案"，那和普通的 RAG 没有区别。赛题的
难点在于⽂档太⻓（⼀篇年报 10 万字），不能全部塞进去，所以系统必须具备⾃主决
策能⼒：判断哪些内容需要精读、哪些可以跳过、读完⼀遍后信息是否⾜够、不够的
话再从哪⾥补。这种"读⼀点、判断⼀下、再决定读不读"的循环，才是 Agent 的核⼼
特征。

A.3 Rerank（重排序）
Rerank 是在初步检索结果之上做⼆次精排的技术。
为什么需要 Rerank？因为初步检索（⽆论是 BM25 还是 Embedding）追求速度，通
常⽤⽐较轻量的算法，结果中可能混⼊⼀些"看起来相关但不精确"的⽚段。Rerank ⽤
更精细的模型（通常是 Cross-Encoder）对 top K 结果逐⼀打分，把真正相关的排在
前⾯。
具体流程：
1. 粗排（召回）：⽤ BM25 或 Embedding 快速从上万⽚段中挑出 50-100 个候选
2. 精排（Rerank）：把"问题 + 每个候选⽚段"⼀起送⼊ Cross-Encoder，模型输出⼀
个相关性分数
3. 取 top K：按分数排序，只把最⾼分的 5-10 个⽚段交给 LLM
Cross-Encoder 的⼯作原理和 Embedding 不同：Embedding 是分别把问题和⽚段转
成向量再⽐较；Cross-Encoder 是把问题和⽚段拼接成⼀段⽂字⼀起输⼊模型，让模
型直接判断"这段⽂字和问题有多相关"。Cross-Encoder 更精确，但速度更慢，所以只
⽤在少量候选上。
搜索引擎（百度、⾕歌、夸克）的搜索结果⻚其实也是这个逻辑：先通过倒排索引快
速召回百万⽹⻚，再⽤复杂的排序算法（考虑了上百个特征）把最相关的排到前⾯。
A.4 BM25
BM25 是⼀种基于词频统计的⽂本检索算法，是信息检索领域最经典、最常⽤的算法
之⼀。它的核⼼思想很简单：如果⼀个词在⽂档中出现次数多、在整体⽂档集中出现
次数少，那么这个词对该⽂档的区分度就⾼。
BM25 计算每个词对⽂档的相关性贡献：
• 词频（TF）：词在⽂档中出现的次数。出现越多，贡献越⼤。但不是线性增⻓（出
现 100 次不代表⽐出现 10 次重要 10 倍），BM25 ⽤⼀个对数函数做饱和处理。
• 逆⽂档频率（IDF）：词在所有⽂档中出现的频繁程度。"的""是"这种词在每篇⽂档
都出现，IDF 很低；"退保费⽤"只在保险条款中出现，IDF 很⾼。
最终得分 = 所有查询词的 TF × IDF 之和。
BM25 的优点：
• 纯统计⽅法，不依赖任何外部模型，计算速度快
• 对精确匹配（如数字、年份、条款编号）很敏感
• 实现简单，jieba 分词 + rank-bm25 库⼏⾏代码就能跑

A.5 Token 与上下⽂窗⼝
Token 是⼤语⾔模型处理⽂本的基本单位。它不是字符，也不是单词，⽽是模型在训
练时从海量⽂本中学到的"词⽚段"。
中英⽂的 Token 换算⼤致如下：
• 1 个英⽂单词 ≈ 1-2 个 Token
• 1 个汉字 ≈ 1-1.5 个 Token（具体取决于模型 tokenizer）
• 1 个标点符号 ≈ 1 个 Token
• 数字通常逐位拆分为 Token（"2025"可能是 2 个 Token："20"+"25"）
举例："中国平安的 2024 年营业收⼊是 1 万亿元"这句话，Qwen 系列模型⼤约需要
15-20 个 Token。
上下⽂窗⼝（Context Window） 是指模型⼀次能处理的 Token 数量上限。Qwen-
plus 从 Qwen-Plus-2025-07-28 快照版本起⽀持 1M Token（约 80 万汉字），默认
128K（约 10 万汉字），可通过 max_input_token 参数调整 [27]。看起来很⻓，但实
际使⽤时有两个约束：
1. 经济约束：prompt 越⻓，消耗的 Token 越多，费⽤越⾼。本赛题 500 万 Token
的总预算，意味着你全部⽤完的话，平均每条 prompt 只能花 5 万个 Token（100 题）。
如果把 10 万字的年报全⽂塞进去，⼀道题就花掉 10 万 Token，100 题就超预算了。
2. 注意⼒衰减：即使模型技术上⽀持 128K 上下⽂，实验表明模型对 prompt 中间部
分的内容记忆效果会下降（称为"中间丢失"问题）。关键信息如果藏在 prompt 中段，
模型可能"视⽽不⻅"。
这就是本赛题"动态记忆压缩"的核⼼挑战：不能⽆脑塞全⽂，必须在"保留⾜够信息"和
"控制 Token 消耗"之间做取舍。

4. 增信措施：担保、抵押、质押等信⽤增进安排
5. 特殊条款：
￮ 回售条款：投资者有权在特定时间按约定价格将债券卖回给发⾏⼈
￮ 赎回条款：发⾏⼈有权在特定时间提前赎回债券
￮ 调整票⾯利率条款：发⾏⼈有权调整后续计息期间的利率
￮ 交叉违约条款：发⾏⼈在其他债务上违约，本债券也视为违约
答题关键：
• 条款判断题要定位到具体条款原⽂，不能凭常识
• 计算题（如回售收益率）要确认时间节点和利率
• 评级相关题要区分"主体评级"和"债项评级"

B.5 ⾏业研报
⾏业研报是券商研究机构发布的对某个⾏业的深度分析，通常包含数据、观点、预测
和图表。
典型结构：
1. ⾏业概况：⾏业定义、产业链、市场规模
2. 竞争格局：主要参与者、市场份额、集中度（CR4、CR8）
3. 驱动因素：政策、技术、需求、供给等影响⾏业发展的关键因素
4. 数据与预测：历史数据、增⻓率、未来预测
5. 投资建议：看好/看平/看空，⻛险提示
答题关键：
• 事实查找题要注意年份限定（"2022 年"和"2023 年"数据可能完全不同）
• 研报中的数据通常有明确来源标注，但模型可能混淆不同来源的数据
• 研报观点（"我们认为..."）和数据事实要区分开，观点不⼀定正确
